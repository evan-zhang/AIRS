# Counter Consensus Analysis（反共识分析）

**归属 Milestone**：M2 Methodology Core  
**适用层级**：Methodology Layer → Evaluation Layer → Report Layer  
**免责声明**：本文档用于训练研究中的反方思考，不构成投资建议，不提供买卖指令或收益承诺。

## 1. Purpose（目的）

反共识分析用于识别市场主流叙事、分析其证据基础，并构造可验证的反方观点。目标不是为了反对而反对，而是检验研究结论是否忽略关键风险、替代解释或边际变化。

## 2. Theory（理论基础）

理论基础来自行为金融、贝叶斯更新和假设检验。市场共识可能因确认偏误、代表性偏差、叙事惯性或信息滞后而失真。反共识研究通过寻找被低估的证据和逻辑断点提高研究鲁棒性。

## 3. Background（背景）

AIRS 要求所有报告包含反方观点。反共识分析为 Evaluation Layer 提供方法，确保 Agent 不只输出顺滑叙事，还能主动质疑需求、供给、利润、估值和政策假设。

## 4. Applicable Scenarios（适用场景）

- 市场已有强烈一致预期或热门主题。
- 研究报告中出现单边乐观或单边悲观判断。
- 用户要求验证某个被广泛接受的观点是否可靠。

## 5. Non-applicable Scenarios（不适用场景）

- 基础事实尚未建立，无法定义共识。
- 用户要求制造耸动观点或无证据反驳。
- 涉及非公开信息或内幕消息推断。

## 6. Inputs（输入要求）

- 待检验的市场共识或主流命题。
- 共识来源：报告、新闻、业绩会、行业数据或市场指标。
- 可用证据集和时间范围。
- 需要检验的关键假设，例如需求增速、利润率、竞争格局或政策持续性。

## 7. Outputs（输出规范）

- 共识命题拆解：事实、假设和外推分别列出。
- 反共识候选观点：每条说明触发条件和证据。
- 证据强度评估：支持证据、反证和缺口。
- 结论更新：维持、削弱、推翻或等待更多证据。

## 8. Workflow（工作流程）

1. 明确共识是什么，以及它依赖哪些核心假设。
2. 为共识命题建立支持证据链。
3. 从数据、逻辑、时间、场景和竞争五个方向寻找反证。
4. 判断反证是噪声、局部例外还是结构性变化。
5. 给出共识修正幅度和置信度。
6. 在报告中保留反方观点与后续验证指标。

## 9. Required Evidence（必需证据）

- 至少一个能代表市场共识的公开来源。
- 支持共识的关键事实和数据。
- 反共识证据，优先使用财报、订单、价格、政策或竞争数据。
- 若反方观点仅为逻辑推演，必须明确标注低置信度。

## 10. Counter Evidence（反证要求）

- 对反共识观点本身也要提出反证，避免把反方观点当作新共识。
- 检查反证样本是否具有代表性。
- 检查反证是否已被市场或原共识吸收。
- 检查时间窗口是否错配。

## 11. Failure Cases（失效场景）

- 把少数异常样本过度外推。
- 因为观点新颖就提高其可信度。
- 未区分“共识错误”和“共识过早或过度”。
- 输出反共识后引导交易行为。

## 12. Confidence（置信度评估方法）

置信度取决于共识定义清晰度、反证质量、样本代表性、可重复性和时间匹配度。若反方证据只能说明短期扰动，应标注为中低置信度，不得推翻长期命题。

## 13. Benchmark Mapping（基准测试映射）

Benchmark 应提供带有主流观点和混合证据的案例，评估 Agent 是否能拆解共识、识别关键假设、提出有证据的反方观点，并避免为了反常识而牺牲事实。

## 14. Future Prompt Mapping（未来 Prompt 映射）

Prompt 应强制输出“共识命题表”“关键假设表”“反方证据表”“对反方观点的再反驳”。最终结论必须以研究参考形式呈现。

## 15. Future Skill Mapping（未来 Skill 映射）

映射到 `committee`、`evidence`、`risk` 和 `report` Skill。Review Agent 可复用该方法审查报告是否存在单边叙事。

## 16. Future Score Mapping（未来评分映射）

评分维度包括共识拥挤度、假设脆弱性、反证质量、反证代表性、结论修正幅度和不确定性披露质量。评分不是逆向交易信号。
