# Risk Analysis（风险分析）

**归属 Milestone**：M2 Methodology Core  
**适用层级**：Methodology Layer → Evaluation Layer → Report Layer  
**免责声明**：本文档仅用于研究风险识别与披露，不构成投资建议、风控承诺或交易指令。

## 1. Purpose（目的）

风险分析用于系统识别研究结论可能失效的条件、影响路径和监测指标。该方法论确保 AIRS 报告不仅有核心观点，还明确数据风险、模型风险、市场风险、认知风险和技术风险。

## 2. Theory（理论基础）

理论基础来自风险管理、情景分析和假设检验。任何研究结论都依赖事实、假设和外部环境；风险分析通过识别关键假设的破坏条件，提高报告透明度和可复核性。

## 3. Background（背景）

M1 Master Skill 已要求所有报告披露风险。M2 风险方法论为后续 Evaluation Engine 和 Report Engine 提供统一风险分类、证据要求和评分口径。

## 4. Applicable Scenarios（适用场景）

- 所有投资研究报告和方法论输出。
- 研究结论依赖需求、政策、技术、财务或估值假设。
- 需要审查报告是否存在单边结论和风险遗漏。

## 5. Non-applicable Scenarios（不适用场景）

- 用户只要求术语解释且不形成研究结论。
- 风险无法被定义或监测，且缺少任何证据来源。
- 用户要求用风险分析规避投资责任或保证收益。

## 6. Inputs（输入要求）

- 核心研究结论和关键假设。
- 支持证据、反证和不确定性列表。
- 研究对象的行业、政策、技术、财务和市场背景。
- 需要输出的风险粒度：简要、标准或详细。

## 7. Outputs（输出规范）

- 风险清单：风险名称、触发条件、影响路径和证据。
- 风险等级：影响程度、发生可能性和可监测性。
- 结论失效条件：哪些变量变化会推翻或削弱结论。
- 跟踪指标、缓释因素和免责声明。

## 8. Workflow（工作流程）

1. 拆解核心结论依赖的关键假设。
2. 按数据、模型、市场、政策、技术、财务和认知风险分类。
3. 为每类风险寻找证据、触发条件和影响路径。
4. 评估风险等级和可监测性。
5. 生成反方情景，说明结论如何失效。
6. 输出风险矩阵和后续跟踪清单。

## 9. Required Evidence（必需证据）

- 支撑风险存在的历史案例、数据、公告、政策或行业报告。
- 与核心结论直接相关的触发变量。
- 风险影响路径证据，而非泛泛风险提示。
- 对风险概率和影响程度的依据说明。

## 10. Counter Evidence（反证要求）

- 是否存在缓释因素，例如长期合同、成本传导、政策保护或技术替代。
- 风险是否已被市场、公司或行业主体充分应对。
- 风险触发条件是否过于极端或缺乏现实基础。
- 风险是否与研究结论无直接关系。

## 11. Failure Cases（失效场景）

- 使用模板化风险提示，没有具体触发条件。
- 只列负面风险，不分析缓释因素。
- 把所有不确定性都列为高风险。
- 风险披露缺少证据和监测指标。

## 12. Confidence（置信度评估方法）

风险结论的置信度由风险证据质量、历史发生频率、触发变量可观测性、影响路径清晰度和缓释因素强度决定。无法观测或纯猜测风险应低置信度标注。

## 13. Benchmark Mapping（基准测试映射）

Benchmark 应检查 Agent 是否能识别关键假设失效条件、区分主要风险和泛化风险，并给出可监测指标。评测不要求预测风险发生时间。

## 14. Future Prompt Mapping（未来 Prompt 映射）

Prompt 应强制输出风险矩阵、触发条件、影响路径、缓释因素和不确定性。最终报告必须包含“不构成投资建议”的风险免责声明。

## 15. Future Skill Mapping（未来 Skill 映射）

映射到 `risk`、`evidence`、`committee`、`financial` 和 `report` Skill。Risk Skill 负责统一风险分类，Report Skill 负责纳入最终披露。

## 16. Future Score Mapping（未来评分映射）

评分维度包括影响程度、发生可能性、证据质量、可监测性、缓释强度、结论相关性和披露完整度。风险评分用于研究质量控制，不构成投资风险承诺。
