# Policy Driven Analysis（政策驱动分析）

**归属 Milestone**：M2 Methodology Core  
**适用层级**：Methodology Layer → Research Skill Layer → Evidence Layer  
**免责声明**：本文档仅用于公开政策研究，不构成法律意见，不构成投资建议，也不提供交易指令。

## 1. Purpose（目的）

政策驱动分析用于研究政策、监管、财政补贴、产业规划和贸易规则如何影响行业供需、竞争格局、成本结构和风险边界。目标是把政策文本转化为可验证的研究假设。

## 2. Theory（理论基础）

理论基础来自公共政策分析、监管经济学和产业政策传导。政策影响通常通过准入、补贴、税收、采购、标准、限制和监管处罚传导，但政策目标、执行力度和地方落地差异会影响实际效果。

## 3. Background（背景）

AIRS 覆盖的新能源、半导体、创新药和 AI 基础设施均受到政策影响。政策驱动方法能避免 Agent 简单把政策表述等同于业绩结果。

## 4. Applicable Scenarios（适用场景）

- 用户询问政策对某行业、公司或产业链的影响。
- 研究对象存在补贴、准入、监管、标准或贸易限制。
- 需要判断政策催化是短期事件还是长期制度变量。

## 5. Non-applicable Scenarios（不适用场景）

- 政策与研究对象没有明确传导路径。
- 用户要求法律合规结论或监管预测。
- 用户要求根据政策直接买卖相关资产。

## 6. Inputs（输入要求）

- 政策文件名称、发布机构、发布时间和适用范围。
- 政策目标、工具、约束和执行主体。
- 涉及行业、企业类型、产品标准或区域。
- 历史政策和执行数据作为对照。

## 7. Outputs（输出规范）

- 政策摘要：目标、工具、约束和时间表。
- 传导路径：政策 → 行业机制 → 企业行为 → 财务或风险影响。
- 受益与受损环节分析，附证据和反证。
- 执行不确定性、风险和免责声明。

## 8. Workflow（工作流程）

1. 获取政策原文并确认适用范围。
2. 拆解政策工具和约束条件。
3. 建立传导路径，区分直接影响和间接影响。
4. 查找历史类比和执行证据。
5. 生成反证，检查落地难度、预算约束和监管变化。
6. 输出影响矩阵、置信度和跟踪指标。

## 9. Required Evidence（必需证据）

- 政策原文或官方解读。
- 实施细则、预算、监管口径或地方执行文件。
- 行业数据或公司公告中体现政策影响的证据。
- 历史政策效果对照。

## 10. Counter Evidence（反证要求）

- 政策是否只有方向性表述，缺少执行细则。
- 预算、地方执行或监管资源是否不足。
- 行业主体是否能通过替代路径规避约束。
- 政策影响是否已被市场或企业提前反映。

## 11. Failure Cases（失效场景）

- 只引用政策标题，不分析执行机制。
- 把政策目标当作确定结果。
- 忽略政策滞后、地方差异和执行成本。
- 将政策催化直接转化为投资建议。

## 12. Confidence（置信度评估方法）

置信度由政策层级、执行细则清晰度、预算或处罚机制、历史执行记录和证据一致性决定。方向性规划通常低于强制性标准或已配套资金的政策。

## 13. Benchmark Mapping（基准测试映射）

Benchmark 应包含强执行政策、弱执行政策、政策反转和市场过度解读案例。评测 Agent 是否能从原文提取机制，而不是复述政策口号。

## 14. Future Prompt Mapping（未来 Prompt 映射）

Prompt 应要求政策原文摘要、工具分类、影响矩阵、执行证据和反证。输出需区分政策方向、政策落地和财务影响。

## 15. Future Skill Mapping（未来 Skill 映射）

映射到 `news`、`industry`、`risk`、`evidence` 和 `report` Skill。News Skill 捕捉政策，Industry Skill 建立传导，Risk Skill 评估监管不确定性。

## 16. Future Score Mapping（未来评分映射）

评分维度包括政策强度、执行确定性、受益直接性、时间滞后、监管风险、反证压力和证据质量。评分仅用于研究优先级和影响解释。
